# scanline 高速化計画 — 計測ゲート方式

ステータス: **計画のみ (2026-06-13)**。着手前に毎ステップ「計測 → 判定 →
採用/revert」を挟む。実装ブランチ: feat/scanline-opt (マイルストーンで
main へマージ)。

## 0. 背景 (実測と結論)

θファン実証実験 (2026-06-13、worktree-scanline-theta-fan → 407dcbd) 時の
テレメトリ: scan **7〜38ms/f**、pv 36〜45ms、loc 4〜9ms (PIE)。

scan の支配項は **PSRAM の散発読み**であって ALU ではない:
- `bc_sample_line` (bc_locate.c) は 16.16 固定小数点 + 最近傍 3 点
  ビニングで、サンプルあたり十数命令と既に軽い
- 領域パス 16 本 × ~800 サンプル × 3 ビン ≈ 38k 回の散発読み。斜め角度
  ではほぼ毎読みが別キャッシュライン (64B) → 実効 ~2.4MB/フレーム
- ean13 デコーダは van Herk + RLE の O(n) で対象外

よって **PIE 直行は筋が悪い** (esp-dsp がテンソルで負けたのと同じ構図、
docs/pie-tensor-asm-plan.md §1)。効くのは「読み方を変える」アルゴリズム
側。また pv (PPA ピクセルレート) がフレーム天井なので、scan 単独の短縮は
fps では 16 → 20 程度が上限 — 狙いは fps ではなく**手ブレ時のロック追従
とレイテンシ**。

## 1. 計測プロトコル (全ステップ共通)

1. cam_demo の連続スキャンモードで、同一の本・同一照明・三脚等で固定:
   - シナリオ A: 正対 (ファン発火なしの基準線)
   - シナリオ B: 5〜10° 傾け保持 (near-miss + ファン経路)
2. 各シナリオ 10 スキャン分の `camera.status()` 末尾
   `[pv loc scan ms/f ... fanR/H]` を記録。取得はスキャン毎 MQTT publish
   を一時的に足す (θファン実験と同じ実験足場 — **コミットしない**)
3. 判定は **scan ms/f の中央値**の前後比較。pv / loc も併記して副作用
   (PSRAM 帯域の取り合い) を監視

## 2. ステップ (効く順、各々に計測ゲート)

### S1. センターアウト走査順 (リスクゼロのウォームアップ)
- scan_region の 16 本を端→端ではなく中央→外の順に。バーコードは領域
  中心に居がちなので、成功フレームの平均試行本数が減る
- 期待: シナリオ A の scan 数 % 〜十数 % 減。~10 行
- ゲート: 悪化さえしなければ採用 (デコード結果は順序不変)

### S2. 領域 bbox ステージング + luma8 化 → **実測で棄却 (2026-06-13)**

実装した (bc_stage_region / bc_sample_line_l8、ホストで全 7 角度 ×
中央/コーナー × ファン角度のビット一致 PASS) 上で、実機の**同一セッション
内 A/B** (`rg<staged>/<direct>us` 一時テレメトリ) で計測した結果:
**staged 中央値 45.4ms vs direct 20.8ms** — 小領域をステージ経由で
処理する方が、より大きい領域 (cap 超過で direct に落ちた側) の直接走査
より 2 倍遅い。敗因:

1. **luma() 変換が ALU 律速** (乗算 3/px ≈ 12 cyc/px → 512K px ≈ 17ms)。
   「シーケンシャルコピーは安い」の見積りはメモリ帯域だけ見ていて
   変換コストを見落としていた
2. **P4 の L2 は 128KB**。256-512KB のステージは L2 に収まらず、
   サンプル読みは結局 PSRAM 散発に戻る = 変換税 + 散発の二重払い。
   ステージが効く条件 (≤100KB) では、散発が一番痛い大領域に使えない —
   構造的に成立しない

プリミティブ自体はホストテスト付きで bc_locate に残置 (L2 内サイズ +
PIE 変換の組で再挑戦する場合のみ触ること)。**派生で採用 (S1.5)**:
walk 長を bbox 対角/2+48 → **バー方向射影/2+48** に短縮 — direct パス
自体が縮む独立改善で、S2 と無関係に有効。

### S3. PIE luma 変換カーネル → S2 棄却に連動して**保留**
(L2 内ステージ + PIE 変換の組合せでのみ意味を持つ。単独では対象なし)

### (旧 S3 原文) PIE luma 変換カーネル (条件付き)
- S2 のステージングコピーをプロファイルし、変換ループが **>1ms** を
  占める場合のみ。`esp.vld.128` → `esp.andq` 0x07E0 → 詰めてストア
  (tensor カーネルの道具立て + 起動 self-check + C fallback を踏襲)
- コピーもメモリ律速なので C -O2 比 2 倍出るか怪しい — だから条件付き
- ゲート: 変換部の実測が半分にならなければ捨てる (保守コストに見合わない)

### S4. PPA 非ブロッキング パイプライン (最大の壁時計効果、最大リスク)
- 現在 pv (PPA 2 パス、ブロッキング) → scan が直列で ~60ms/f。PPA を
  non-blocking にして「フレーム N の PPA」と「フレーム N-1 の scan」を
  重ねれば scan はほぼ隠れ、~42ms/f ≈ 22fps
- 要 s_mid / preview のダブルバッファ (PSRAM +1MB 級) と完了同期
- **PSRAM 飽和の再来リスク** (CSI 115 + DSI 110 MB/s 常時の上に PPA と
  CPU 散発読みが同時に乗る)。c2e4e6a で踏んだ教訓どおり、pv ms/f の
  悪化・プレビューのカクつきを飽和兆候として監視
- ゲート: pv 中央値が 10% 以上悪化したら即 revert (バスを買い戻せない)

### 不採用 (理由つき)
- 任意角サンプリング本体の PIE 化: P4 PIE に gather がない
- ean13 デコーダ最適化: O(n) で軽く、メモリにも優しい
- 半解像度 (s_mid) からのデコード: 細バーのモジュール幅が 1px を切って
  消える (locate 用途と要求解像度が違う)

## 3. 完了条件 / やめ時

- 正対ロック時 scan ≤ **5ms/f** に到達、または S2 完了時点で追従性に
  体感不満が残らなければ S3/S4 はやらない (pv 天井がある以上、scan を
  ゼロにしても fps は ~22 止まり)
- 各ステップは独立コミット + 計測値をコミットメッセージに残す
