# mqjs ランタイム CPU ホットパス監査と修正候補

ステータス: **監査済み・修正は未着手**(2026-06-13、コミット da2a517 時点)。
手法: バインディング実装の読解 + 実バイナリの逆アセンブル +
camera.status() テレメトリの実測。推測には (推) を付す。

前提となる構造:
- **JS タスク = Core0 / prio5 / 全アプリ協調シングルスレッド**。
  ここを 1ms 止めると全アプリのタッチ応答が 1ms 遅れる。
- LVGL タスク = Core1。cam_scan / http_get = prio4・コア未指定。
- UI 描画は 16B コマンド × 深さ 128 のキュー経由(溢れはドロップ+
  ステータスバーにカウンタ表示)。イベントキューは深さ 32。

## 1. Tier 1 — JS タスクを直接ブロックする

### 1.1 store.set / store.del の同期 nvs_commit 【最優先】

```c
/* js_store_set (mqjs_runtime.c) */
bool ok = nvs_set_str(s_store, key, val) == ESP_OK &&
          nvs_commit(s_store) == ESP_OK;   /* フラッシュ書込を JS 上で待つ */
```

- コスト: フラッシュ書込 数 ms〜。NVS ページ GC 発生時は数十 ms 超 (推)。
- 頻度: reading.js は 1 記録ごとに save() = **commit×2**。circuit.js は
  部品 1 配置・セル 1 確定ごと。
- 影響: 全アプリ停止 + フラッシュ摩耗。
- **修正候補**: write-behind(ライトコアレッシング)。
  `nvs_set_*` は即時(RAM キャッシュ)、commit は dirty フラグ +
  300ms 程度のデバウンスタイマー(esp_timer、JS タスクにイベントで
  戻して commit するか、専用低優先タスク)。電源断ウィンドウが
  デバウンス幅だけ生じる点は store.flush() の追加 API か、
  アプリ側重要操作(読了など)での即時 commit オプションで補う。

### 1.2 mquickjs compacting GC のポーズ 【計測手段がない】

- scene() 型の全画面再描画は 1 回で一時文字列 50〜100 個を生成。
  256KB アリーナが埋まるたびに compaction(コスト ∝ live set)(推)。
- **修正候補(まず可視化)**: sys.heap() に GC 回数・累計 µs を追加
  (エンジンにフックがあるか要調査。なければ JS_Run/JS_Call 前後の
  アリーナ水位で代替観測)。数字が出てから対策を判断。

### 1.3 http.get コールバックの一括 48KB 文字列

- C 側で最大 48KB を malloc → JS_NewStringLen で**アリーナへ一括コピー**
  (GC 誘発)→ JS 正規表現で複数回線形走査。
- ワンショット(NDL 検索)では許容。**ポーリングや大きい API への
  使い回しは非推奨**。修正方向は §5 のストリーミング項を参照。

### 1.4 delay(ms) — 既知・文書化済みの全停止。現状のままで良い。

## 2. Tier 2 — バインディング per-call コスト(頻度で効く)

### 2.1 ui.text / ui.cells: 呼び出しごとの malloc

```c
char *copy = malloc(len + 1);   /* internal heap、毎回 */
memcpy(copy, str, len);
ui_post(UI_CMD_TEXT, ..., copy); /* 消費側で free */
```

- 1 call ≈ ToCString + malloc + memcpy + xQueueSend ≈ 数 µs (推)。
  circuit の scene() ≈ 70 call/キーストローク ≈ 0.5ms — 現状無害。
- **60fps × 数百 call の全画面キャンバスアプリが現れたら ms 級**。
- **修正候補**: ui_cmd_t に短文(〜8B 程度)をインライン格納し、
  malloc は長文のみに。互換 API のままで済む。

### 2.2 print / console.log の 1 バイトループ

- out_write() は出力を 1 文字ずつ行アセンブラへ積む(UTF-8 境界
  処理つき)。ANSI アニメ系(life.js)は数 KB/フレーム → ms 級 (推)。
- **修正候補**: memchr('\n') で改行まで塊 memcpy。境界処理は
  行頭/行末だけに残す。~10 倍 (推)。

### 2.3 イベントディスパッチ自体 — 問題なし(計測根拠つき)

- JS_Call 準備 + arm_watchdog(esp_timer 読み 1 回)= µs 級。
  タッチ move 60Hz でも余裕。手を入れない。

## 3. Tier 3 — 別タスクの共有資源競合

### 3.1 cam_scan / http_get タスクのコア未指定

- prio4 なので JS(prio5/Core0)には勝てないが、**Core1 の LVGL と
  競合し得る**(スキャン中の描画ジッタの一因 (推))。
- **修正候補**: `xTaskCreatePinnedToCore(..., 0)` に変更(各 1 行)。
  JS の隙間時間に押し込まれ、LVGL が安定する。

## 4. Tier 4 — キュー溢れ(静かな取りこぼし)

- UI キュー 128 深: scene() 70 件は OK。ドロップはカウンタ表示あり。
- **イベントキュー 32 深**: タッチ move + MQTT 流入の重なりで
  ev_post が黙って捨てる(入力イベントは再送されない)。
- **修正候補**: 32 → 64(+1.3KB internal、実質タダ)。

## 5. 設計論: 一括確保 vs ストリーミング/スライディングウィンドウ

「IO はストリーミング、フィルタはスライディングウィンドウ」という
定石への所見。**結論: 流れるデータには既に使っている。残っている
一括確保は『上限で縛った有界データ』であり、それは定石のもう一方の
解(bound + batch)。** 個別に:

| 経路 | 現状 | 評価 |
|---|---|---|
| カメラフレーム | V4L2 リングバッファ(2 面の滑り窓)+ 走査線単位処理。グレースケール全面画像は**作らない** | ストリーミング済み |
| ean13 二値化 | van Herk の**スライディング min/max**(窓 64、O(n)) | 定石そのもの |
| print 行アセンブラ | ストリーム処理(ただし 1 バイト粒度 → §2.2) | 粒度だけ直す |
| MQTT ペイロード | 4KB 上限の heap コピー + 所有権付きイベント | 有界 batch で正解 |
| NVS 値 | 3.9KB 上限 | 有界 batch で正解 |
| PPA 中間画像 (s_mid) | 一括確保 | **ハードの制約**: PPA はフレーム単位でしか動けない。クロップで「窓」を空間方向に縮めたのが今回の最適化 |
| **http.get body** | **48KB 一括 → JS 文字列一括** | ここだけ将来ストリーミング化の価値あり(下記) |

### http のストリーミング化(将来案)

`http.stream(url, fn(chunk, done))` を別 API として追加(get は互換
維持)。C 側は 4KB チャンクごとに EV_HTTP_CHUNK を発行。
- 利点: アリーナの 48KB スパイク消滅、最初のデータまでの遅延減、
  上限撤廃(大きな API も読める)。
- 代償: **チャンク境界をまたぐマッチ**という古典的問題が JS 側に移る。
  NDL のような XML なら「最後の `</item>` 以降の尻尾を持ち越す」
  持ち越しバッファ(まさにスライディングウィンドウ)で解けるが、
  アプリ作者ごとにこの状態機械を書かせることになる。
- 判断: 取得物が有界(〜48KB)で頻度が低いうちは現行 get で十分。
  RSS リーダー等「大きい/頻繁」の種が出たら stream を足す。

### JS 側アプリ作者向けの注意(ドキュメント行き)

- mquickjs の文字列は不変。ループ内 `s += x` の積み上げは O(n²)。
  大きな出力は配列に push して最後に join(これも streaming の
  心: 中間結果を固めない)。

## 6. 推奨着手順

1. §1.1 store.set の commit 遅延(体感とフラッシュ寿命に直結)
2. §1.2 GC 可視化(sys.heap 拡張)→ 数字を見てから対策判断
3. §2.2 out_write 塊コピー化(安い)
4. §3.1 コアピン留め 2 行 + §4 キュー 64 化
5. §2.1 ui バインディング malloc レス化は高 fps キャンバスアプリの
   種が現れてから
6. §5 http.stream は需要が出てから

計測インフラ: camera.status() の "[pv loc scan ms/f]" 方式
(スキャン終了時に集計値を文字列で出す)が今回有効だったので、
JS ディスパッチにも同様の per-app 累計 µs カウンタを足して
sys.heap() あたりから読めるようにすると、以後の監査が常時できる。
